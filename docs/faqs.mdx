## Getting Started

### What is Continue?

Continue runs AI-powered code review checks on every pull request. Each check is a markdown file in your repository (`.continue/checks/`) that reviews your code changes and reports results as GitHub status checks — green for pass, red for fail with a suggested fix.

### How do checks run?

Checks run in CI via a GitHub Action. When a PR is opened:
1. The GitHub Action triggers
2. The `continue/check` CLI reads your `.continue/checks/` files
3. The AI reviews the PR diff against each check
4. Results are posted as GitHub status checks

See [Run Checks in CI](/checks/running-in-ci) for setup instructions.

### What happens when a check fails?

The PR shows a red ❌ status on GitHub. Continue posts a comment with:
- The failing check's name
- An explanation of what was flagged
- An AI-generated fix suggestion

You can accept the fix directly from GitHub or request changes.

### Is my code/data private?

By default, code is sent to the AI model you configure. Options for privacy:
- **Self-host models**: Use Ollama, LM Studio, or any self-hosted provider
- **Privacy-conscious providers**: Anthropic, OpenAI, etc. have strong data policies
- **Offline setup**: Run Continue completely without internet. See [Running Continue Without Internet](/guides/running-continue-without-internet)

### How do I add Continue to my project?

1. Install the [VS Code](/ide-extensions/install) or [JetBrains](/ide-extensions/install) extension
2. Create a check file in `.continue/checks/`
3. Connect to GitHub via the [CI setup](/checks/running-in-ci)

That's it — checks run automatically on every PR.

## Networking Issues

### Configure Certificates

If you're seeing a `fetch failed` error and your network requires custom certificates, configure them in your model config. Add `requestOptions.caBundlePath`:

<Tabs>
  <Tab title="YAML">
    ```yaml
    models:
      - name: My Model
        ...
        requestOptions:
          caBundlePath: /path/to/cert.pem
    ```
  </Tab>
  <Tab title="JSON">
    ```json
    {
      "models": [
        {
          "name": "My Model",
          "...": "...",
          "requestOptions": {
            "caBundlePath": "/path/to/cert.pem"
          }
        }
      ]
    }
    ```
  </Tab>
</Tabs>

You can also set `requestOptions.caBundlePath` to an array for multiple certificates.

<Warning>
  **Windows VS Code Users:** The [win-ca extension](https://marketplace.visualstudio.com/items?itemName=ukoloff.win-ca) may help Continue use the Windows certificate store, but `requestOptions.caBundlePath` is the most reliable fix.
</Warning>

### Common SSL certificate errors

Errors like `unable to verify the first certificate`, `self signed certificate in certificate chain`, or `CERT_UNTRUSTED` mean Continue reached the endpoint but couldn't verify its TLS certificate chain.

The fix is to export the root or intermediate CA certificate and set `requestOptions.caBundlePath`. For step-by-step diagnosis with `curl` and `openssl`, see the [SSL troubleshooting guide](/troubleshooting#ssl-certificate-errors).

### VS Code Proxy Settings

VS Code handles proxy settings automatically via its [Proxy Server Support](https://code.visualstudio.com/docs/setup/network#_proxy-server-support). To verify it's enabled, use `cmd/ctrl` + `,` to open settings and search for "Proxy Support". Unless it is set to "off", VS Code is handling the proxy.

### code-server

Continue works with [code-server](https://coder.com/), but if you see "This is likely because the editor is not running in a secure context", see [code-server's guide on securely exposing the server](https://coder.com/docs/code-server/latest/guide#expose-code-server).

## Changes to Configs Not Showing in VS Code

If you've edited your config but don't see changes in Continue:

1. Open the command palette (`cmd/ctrl` + `shift` + `P`)
2. Type "Reload Window"
3. Select the reload option

This restarts VS Code and reloads all extensions, including the latest config.

## I installed Continue, but don't see the sidebar window

The Continue window appears on the left side by default. You can reveal it by:
- Pressing `cmd/ctrl` + `L`
- Clicking the button in the top right of VS Code to open the right sidebar

## I'm getting a 404 error from OpenAI

If your API key and model are valid but you're still getting 404 errors, you may need to add credits to your [OpenAI billing account](https://platform.openai.com/settings/organization/billing/overview). Try adding $1 and testing again.

## I'm getting a 404 error from OpenRouter

Models that don't support function calling will return 404 errors to Continue. Example: `HTTP 404 Not Found from https://openrouter.ai/api/v1/chat/completions`

## Agent Mode & Tool Issues

If Agent mode is grayed out or tools aren't working, this is usually a model capability configuration issue.

<Info>
  Continue uses system message tools as a fallback for models without native tool support, so most models should work with Agent mode automatically.
</Info>

### Check if your model has tool support

1. Not all models support native tool/function calling, but Continue will automatically use system message tools as a fallback
2. Try adding `capabilities: ["tool_use"]` to your model config to force tool support
3. Verify your provider supports function calling

### Images not uploading

If you can't upload images:

1. Add `image_input` to capabilities
2. Ensure your model supports vision (e.g., GPT-4V, Claude 3, etc.)
3. Check that your provider passes through image data

### Verify current capabilities

- Check mode selector tooltips — they indicate if tools are available
- Try uploading an image — if disabled, the model lacks `image_input`
- Agent mode availability requires `tool_use`

See [Model Capabilities guide](/customize/deep-dives/model-capabilities) for details.

## Android Studio - "Nothing to show" in Chat

Select `Actions > Choose Boot Runtime for the IDE`, then select the latest version, and restart Android Studio. [See this thread](https://github.com/continuedev/continue/issues/596#issuecomment-1789327178) for details.

## I received a "Codebase indexing disabled" notification

This happens on Linux systems that lack AVX2 and FMA CPU features, which LanceDB requires for vector search. Most Continue features will still work normally, but commands like `@codebase`, `@files`, and `@folder` will be disabled.

See [LanceDB issue #2195](https://github.com/lancedb/lance/issues/2195) for details.

## Ollama Issues

For a comprehensive guide, see the [Ollama Guide](/guides/ollama-guide).

### Unable to connect to local Ollama

1. **Verify Ollama is running:** Open http://localhost:11434 — you should see "Ollama is running"
2. **Start Ollama properly:** Use `ollama serve` (not just `ollama run model-name`)
3. **Check your config:** Ensure your `config.yaml` has the correct setup:

```yaml
models:
  - name: llama3
    provider: ollama
    model: llama3:latest
```

### Connection failed to remote Ollama

1. **Configure Ollama to listen on all interfaces:**
   - Set environment variable: `OLLAMA_HOST=0.0.0.0:11434`
   - For systemd: Edit `/etc/systemd/system/ollama.service` and add:
     ```
     Environment="OLLAMA_HOST=0.0.0.0:11434"
     Environment="OLLAMA_ORIGINS=*"
     ```
   - Restart: `sudo systemctl restart ollama`

2. **Update your Continue config:**
```yaml
models:
  - name: llama3
    provider: ollama
    apiBase: http://192.168.1.136:11434  # Your server IP
    model: llama3:latest
```

3. **Check firewall settings:** Ensure port 11434 is open

### Ollama not working in WSL

**Windows 11 22H2+ (Recommended):**
Create or edit `%UserProfile%\.wslconfig`:
```ini
[wsL2]
networkingMode=mirrored
```
Then restart WSL: `wsl --shutdown`

**Older Windows/WSL:**
```powershell
# Add firewall rules
New-NetFireWallRule -DisplayName 'WSL Ollama' -Direction Inbound -LocalPort 11434 -Action Allow -Protocol TCP
New-NetFireWallRule -DisplayName 'WSL Ollama' -Direction Outbound -LocalPort 11434 -Action Allow -Protocol TCP

# Get WSL IP (run 'ip addr' in WSL)
# Add port proxy (replace <WSL_IP>)
netsh interface portproxy add v4tov4 listenport=11434 listenaddress=0.0.0.0 connectport=11434 connectaddress=<WSL_IP>
```

### Docker container can't connect to host Ollama

**Windows/Mac:** Use `host.docker.internal`:
```yaml
models:
  - name: llama3
    provider: ollama
    apiBase: http://host.docker.internal:11434
    model: llama3:latest
```

**Linux:** Use the Docker bridge IP (usually `172.17.0.1`):
```yaml
models:
  - name: llama3
    provider: ollama
    apiBase: http://172.17.0.1:11434
    model: llama3:latest
```

## Local Config

### Managing Secrets and Environment Variables

For running completely offline, see [Running Continue Without Internet](/guides/running-continue-without-internet).

#### Reference secrets in config.yaml

Use `${{ secrets.SECRET_NAME }}` anywhere in your `config.yaml`:

```yaml
models:
  - provider: openai
    apiKey: ${{ secrets.OPENAI_API_KEY }}
```

#### Where secrets are resolved from (in order)

1. **Workspace `.env`:** `<workspace-root>/.env`
2. **Workspace Continue `.env`:** `<workspace-root>/.continue/.env`
3. **Global `.env`:** `~/.continue/.env`
4. **Process environment variables**

<Warning>
  **IDE extensions (VS Code, JetBrains) cannot read shell environment variables.** Setting `export OPENAI_API_KEY=...` in your terminal won't work. You must use a `.env` file.
</Warning>

#### Creating `.env` files

Create a `.env` file in one of the locations above:

```
OPENAI_API_KEY=sk-...
ANTHROPIC_API_KEY=sk-ant-...
CUSTOM_API_URL=https://api.example.com
```

- **Never commit `.env` files** — add them to `.gitignore`
- No quotes around values
- Restart your IDE after changes

### Using Model Addons Locally

Reference pre-configured models from Continue Mission Control using the `uses:` syntax:

```yaml
name: My Local Config
version: 0.0.1
schema: v1
models:
  - uses: ollama/llama3.1-8b
  - uses: anthropic/claude-sonnet-4-6
  - uses: openai/gpt-4
```

Override addon settings:
```yaml
models:
  - uses: ollama/llama3.1-8b
    override:
      apiBase: http://192.168.1.100:11434
      roles:
        - chat
        - autocomplete
```

2. Uninstall Continue

## MCP Server connection issues

### "spawn ENAMETOOLONG" error on macOS

This happens when the environment is too large. **Workaround:** Use the full path to the command instead of relying on PATH:

```yaml
mcpServers:
  - name: Memory MCP server
    command: /usr/local/bin/npx  # Full path instead of just "npx"
    args:
      - -y
      - "@modelcontextprotocol/server-memory"
```

Find the full path: `which npx` (or `which docker`, `which uv`, etc.)

This issue is tracked in [#7870](https://github.com/continuedev/continue/issues/7870) and [#6699](https://github.com/continuedev/continue/issues/6699).

## SSL Certificate Errors

If Continue can reach your model endpoint but can't verify its TLS certificate chain:

**Errors you might see:**
- `unable to verify the first certificate`
- `self signed certificate in certificate chain`
- `certificate verify failed`
- `CERT_UNTRUSTED`

This usually happens with self-hosted models, enterprise proxies, or internal endpoints with private CAs.

### Quick fix: trust the CA bundle

```yaml
models:
  - name: Secure endpoint
    provider: openai
    model: gpt-4.1
    apiBase: https://llm.example.com/v1
    requestOptions:
      caBundlePath: /path/to/ca-chain.pem
```

### Diagnose the problem

1. Reproduce the error to identify the failing hostname
2. Test with `curl`:
   ```bash
   curl -v https://llm.example.com/v1/models
   ```
3. Inspect the certificate chain:
   ```bash
   openssl s_client -showcerts -connect llm.example.com:443 -servername llm.example.com </dev/null
   ```
4. Save the required certificate as a PEM file and set `caBundlePath`
5. Verify the fix:
   ```bash
   curl --cacert /path/to/ca-chain.pem https://llm.example.com/v1/models
   ```

### Common setups

- **Private CA:** Set `caBundlePath`
- **Mutual TLS:** Set `caBundlePath` and `clientCertificate`
- **Windows VS Code:** Try the [win-ca extension](https://marketplace.visualstudio.com/items?itemName=ukoloff.win-ca)

<Tip>
  As a temporary debug step, set `verifySsl: false`. Do not leave this disabled in production.
</Tip>
